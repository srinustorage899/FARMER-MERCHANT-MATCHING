▶ 🧮 Step 1: Vector Encoding (How Query Becomes Numbers)

Both farmer listings AND merchant queries are converted into a fixed-length number array (vector) so we can measure "distance" between them mathematically.

Why normalise? Raw lat/lon are 0–180 but price is 0–500. Without normalisation, price would dominate distance. Normalising to [0,1] ensures every feature is equally weighted.

Why one-hot crop? "Tomato"=1 at position 0, "Onion"=1 at position 1, etc. This ensures the HNSW only finds farmers selling the SAME crop — different crop = huge L2 distance.

Why two phases? Checking every farmer's exact distance is O(n) — slow with millions of records. HNSW narrows candidates to ~50 in O(log n), then exact filtering is cheap on this small set.

▶ 🍀 Step 3: Why These Farmers? (Composite Ranking)

Final ranking uses a composite score combining BOTH distance and price:

score = 0.6 × distance_norm + 0.4 × price_norm

distance_norm = farmer_distance / max_distance  (0 = closest, 1 = farthest)
price_norm = (farmer_price − min_price) / price_range  (0 = cheapest, 1 = priciest)

⚠ Lower score = better match. A farmer who is both nearby AND cheap gets the lowest score.

—

Why both? Using only distance might pick a nearby but expensive farmer. Using only price might pick a cheap but far-away farmer. The composite score balances both — a merchant gets the best value for proximity + cost.

▶ 📚 How HNSW Search Works

HNSW = Hierarchical Navigable Small World

Think of it like an express elevator system in a tall building:

1. Layer 2 (Top — Sparse): Only a few "express" nodes. The query enters here and quickly finds the approximate region.

2. Greedy traversal: At each node, check all connected neighbours. Move to whichever neighbour is CLOSEST to the query vector (lowest L2 distance). Repeat until no neighbour is closer.

3. Descend to Layer 1: Drop down to a denser layer, keeping the same node as starting point. More neighbours = finer search.

4. Descend to Layer 0 (Bottom — Dense): Every farmer exists here. Perform a final refined local search among all connections.

5. Result: The nearest node at Layer 0 is the approximate nearest neighbour.

Why is this fast? Instead of comparing the query against ALL farmers (O(n)), HNSW navigates through a graph in O(log n) steps. With 100,000 farmers, brute force needs 100,000 comparisons; HNSW needs ~17.

L2 distance formula: d = √((x₁-x₂)² + (y₁-y₂)² + ... ) — Euclidean distance across all vector dimensions. Smaller = more similar.

Parameter M=16: Each node connects to up to M neighbours per layer. Higher M = more accurate but slower insertion.

Parameter ef_search=100: Size of the dynamic candidate list during search. Higher = more accurate but slower.

▶ 📏 L2 vs Haversine vs Composite Score

Phase 1 — Inside HNSW: Uses L2 (Euclidean) distance on the normalised vector. This considers location + price + quantity + crop type ALL TOGETHER. It's an approximation used for speed.

Phase 2 — Exact Distance: Uses Haversine distance — the real-world GPS distance in kilometres between merchant and each farmer.

Phase 3 — Composite Score: Combines normalised distance (60% weight) + normalised price (40% weight) into a single score. This ensures the final ranking favours farmers who are both nearby AND affordable.

Analogy: L2 is the rough compass. Haversine is the accurate GPS. Composite score is the final decision — like choosing a restaurant that's both close AND has good prices, not just the closest one.
